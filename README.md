# E-IMZO ATB Client

Локальный Windows-сервис, который поднимает `wss` сервер на `127.0.0.1:64443`, принимает подключения от сайтов или веб-приложений, вызывает локальный E-IMZO API и возвращает обратно тело ответа без изменений.

## Конфигурация через `.env`

Программа автоматически читает настройки из файла `.env`.

Поддерживаются такие сценарии:

- если `.env` лежит в текущей рабочей папке, он будет загружен;
- если приложение собрано в `.exe`, оно также может читать `.env` рядом с исполняемым файлом;
- если задана переменная `APP_ENV_FILE`, будет использован путь из нее.

Приоритет такой:

- системные переменные окружения Windows не перезаписываются значениями из `.env`;
- `.env` работает как источник настроек по умолчанию.

Пример `.env`:

```dotenv
API_EIMZO_URL=http://127.0.0.1:64646
WS_SERVER_HOST=127.0.0.1
WS_SERVER_PORT=64443
WS_SERVER_PATH=/
LOG_LEVEL=INFO
```

В репозитории добавлен шаблон: `.env.example`.

## Что делает программа

Приложение работает как локальный WebSocket bridge между браузером и E-IMZO API.

Схема работы такая:

1. сайт в браузере подключается к локальному `wss://127.0.0.1:64443`;
2. сайт отправляет JSON-команду с `plugin`, `name` и опционально `arguments`;
3. локальное приложение вызывает соответствующий endpoint E-IMZO API;
4. в HTTP-запрос добавляется заголовок `x_account_name` с именем текущей Windows-учетной записи;
5. тело ответа от E-IMZO API отправляется обратно в WebSocket без модификации.

## Поток работы

### 1. Старт приложения

При запуске программа:

1. читает конфигурацию из переменных окружения и `.env`;
2. настраивает логирование;
3. определяет имя текущего Windows-пользователя;
4. создает или продлевает локальный сертификат для `127.0.0.1`;
5. поднимает локальный `wss` сервер;
6. если сертификат перевыпускается в фоне, сервер автоматически перезапускается и подхватывает новый сертификат.

### 2. Прием WebSocket-подключений

По умолчанию сервер слушает:

```text
wss://127.0.0.1:64443/
```

Путь тоже настраивается, но по умолчанию это `/`.

Если подключение приходит не на ожидаемый путь, сервер его отклоняет.

### 3. Обработка входящего сообщения

Ожидается JSON такого вида:

```json
{
  "plugin": "pfx",
  "name": "list_all_certificates"
}
```

Если `arguments` отсутствует, выполняется `GET`:

```text
GET {API_EIMZO_URL}/pfx/list_all_certificates
```

Если `arguments` присутствует, выполняется `POST`, а `arguments` отправляется как JSON-тело:

```json
{
  "plugin": "pfx",
  "name": "load_key",
  "arguments": {
    "disk": "C:"
  }
}
```

```text
POST {API_EIMZO_URL}/pfx/load_key
```

Правила простые:

- `plugin` и `name` напрямую подставляются в путь;
- без `arguments` идет `GET`;
- с `arguments` идет `POST`;
- `x_account_name` передается в каждом HTTP-запросе.

### 4. Возврат ответа

После вызова E-IMZO API программа:

- читает body HTTP-ответа;
- не меняет JSON, текст или бинарные данные;
- отправляет обратно только body.

Это значит:

- HTTP status code отдельно не возвращается;
- HTTP headers отдельно не возвращаются;
- в WebSocket уходит только тело ответа.

## Работа с локальным сертификатом

Программа автоматически управляет локальным сертификатом для `127.0.0.1` и `localhost`:

- создает сертификат, если его нет;
- хранит сертификат и ключ в runtime-папке;
- периодически проверяет срок действия;
- перевыпускает сертификат заранее;
- после перевыпуска перезапускает локальный `wss` сервер, чтобы он начал использовать новый сертификат.

Путь по умолчанию:

```text
%LOCALAPPDATA%\E-IMZO-ATB-Client\certs
```

Если `API_EIMZO_URL` указывает на `https://127.0.0.1` или `https://localhost`, этот же локальный сертификат используется как доверенный CA для обращения к локальному API.

Важно для браузера:

- просто сгенерировать сертификат недостаточно;
- браузер должен доверять этому сертификату, иначе подключение к `wss://127.0.0.1:64443` будет блокироваться;
- если нужно, следующим шагом можно добавить автоматическую установку сертификата в хранилище доверенных сертификатов Windows для текущего пользователя.

## Структура проекта

### `src/client/bootstrap`

Стартовые вещи:

- конфигурация;
- загрузка `.env`;
- базовая настройка логирования.

### `src/client/domain`

Минимальные модели приложения:

- входящая WebSocket-команда.

### `src/client/system`

Интеграция с локальной системой:

- получение имени Windows-пользователя;
- сертификаты;
- runtime-файлы.

### `src/client/integrations`

Интеграции с внешними сервисами:

- вызов локального E-IMZO API;
- формирование HTTP-запросов;
- возврат body ответа.

### `src/client/transport/websocket`

WebSocket-транспорт:

- запуск локального `wss` сервера;
- прием входящих подключений;
- прием сообщений;
- отправка ответов обратно в подключение.

### Точки входа

- `src/client/main.py` — запуск процесса;
- `src/client/app.py` — сборка приложения и orchestration.

## Переменные окружения

### Обязательные

| Переменная | Описание |
| --- | --- |
| `API_EIMZO_URL` | Базовый URL локального E-IMZO API, например `http://127.0.0.1:64646` |

### Локальный WebSocket сервер

| Переменная | По умолчанию | Описание |
| --- | --- | --- |
| `WS_SERVER_HOST` | `127.0.0.1` | Хост, на котором слушает локальный `wss` сервер |
| `WS_SERVER_PORT` | `64443` | Порт локального `wss` сервера |
| `WS_SERVER_PATH` | `/` | Ожидаемый WebSocket path |
| `WS_SERVER_RETRY_DELAY_SECONDS` | `5` | Пауза перед повторным запуском сервера после ошибки |
| `WS_PING_INTERVAL_SECONDS` | `20` | Интервал ping |
| `WS_PING_TIMEOUT_SECONDS` | `20` | Таймаут ожидания pong |

### HTTP и runtime

| Переменная | По умолчанию | Описание |
| --- | --- | --- |
| `HTTP_TIMEOUT_SECONDS` | `60` | Общий таймаут HTTP-запроса в E-IMZO API |
| `LOG_LEVEL` | `INFO` | Уровень логирования |
| `APP_RUNTIME_DIR` | `%LOCALAPPDATA%\E-IMZO-ATB-Client` | Папка для runtime-файлов |
| `APP_ENV_FILE` | - | Явный путь к `.env` файлу |
| `ACCOUNT_NAME_OVERRIDE` | - | Явно задает значение для `x_account_name` |

### Сертификаты

| Переменная | По умолчанию | Описание |
| --- | --- | --- |
| `LOCAL_CERT_VALID_DAYS` | `365` | Срок действия локального сертификата |
| `LOCAL_CERT_RENEW_BEFORE_DAYS` | `30` | За сколько дней до окончания перевыпускать сертификат |

## Пример запуска

Установка зависимостей:

```powershell
pip install -e .
```

Минимальный запуск:

```powershell
eimzo-atb-client
```

Если `API_EIMZO_URL` уже лежит в `.env`, отдельно задавать его в консоли не нужно.

Запуск с явным path:

```powershell
$env:WS_SERVER_PATH = "/eimzo"
eimzo-atb-client
```

После этого сайт может подключаться, например, к:

```text
wss://127.0.0.1:64443/
```

или:

```text
wss://127.0.0.1:64443/eimzo
```

в зависимости от конфигурации.

## Пример сценария

Сайт отправляет:

```json
{
  "plugin": "pfx",
  "name": "list_all_certificates"
}
```

Программа выполняет:

```text
GET http://127.0.0.1:64646/pfx/list_all_certificates
```

С заголовком:

```text
x_account_name: DOMAIN\User
```

Если сайт отправляет:

```json
{
  "plugin": "pfx",
  "name": "load_key",
  "arguments": {
    "disk": "C:"
  }
}
```

Программа выполняет:

```text
POST http://127.0.0.1:64646/pfx/load_key
```

С телом:

```json
{
  "disk": "C:"
}
```

И затем возвращает в WebSocket ровно тот body, который пришел от E-IMZO API.

Он:

- публикует artifact `eimzo-atb-client-windows`;
- при теге вида `v1.0.0` прикладывает `eimzo-atb-client.exe` к GitHub Release.
